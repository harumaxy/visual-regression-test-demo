# VRT（Visual Regression Testing）のおさらい

## 用語と定義

**Visual Regression Testing（ビジュアルリグレッションテスト）**
UIの「見た目」が意図せず変わっていないかを、画像比較によって検証するテスト手法。日本語では「ビジュアル回帰テスト」「視覚的回帰テスト」とも呼ばれます。

**Regression（リグレッション / 回帰）**
もともと動いていた・正しかったものが、変更によって**壊れて元に戻ってしまうこと**。「回帰バグ」「デグレ（degrade）」と同義です。VRTの「R」はこの意味で、「見た目の意図しない後退」を検出するのが目的です。

**Baseline（ベースライン）← ご質問の部分**
**「正しい状態」として基準にする画像**のこと。「これが正解の見た目です」という参照用スクリーンショット。後続のテストはこのベースラインと比較され、差分があればリグレッション候補として検出されます。

「ゴールデンイメージ（golden image）」「リファレンス画像（reference image）」とも呼ばれ、同じ意味です。

**Snapshot（スナップショット）**
ある時点で撮影したUIの画像。文脈で意味が変わります。
- 「ベースラインスナップショット」= 基準として保存されている画像
- 「テストスナップショット」「アクチュアル」= 今回のテスト実行で撮影した画像

**Diff（ディフ / 差分画像）**
ベースラインとテストスナップショットを比較して生成される、**差分を可視化した画像**。変わったピクセルが赤色などでハイライトされます。

**Threshold（しきい値）**
「どこまでの差分なら同じとみなすか」の許容度。完全一致を求めるとフォントレンダリングの揺らぎで毎回失敗するので、現実には数%の差分を許容します。
- `maxDiffPixelRatio: 0.01` → 1%までのピクセル差分は許容
- `maxDiffPixels: 100` → 100ピクセルまでの差分は許容

**Flaky（フレイキー / 不安定なテスト）**
同じコードでも実行のたびに結果が変わるテスト。VRTは特にflakyになりやすく、原因はアニメーション・日付・乱数・フォント描画・ネットワーク遅延など。

**False Positive（フォルスポジティブ / 偽陽性）**
実際にはバグじゃないのに「リグレッションだ」と誤検知されること。VRTで最も嫌われる現象で、これが多いとチームがテストを信頼しなくなります。

**Masking（マスキング）**
動的に変わる領域（日付、アバター、広告など）を塗りつぶして比較対象から外すこと。

**Approve / Accept（承認）**
検出された差分が「意図した変更」だった場合に、新しい画像を**新しいベースラインとして採用する**操作。「ベースラインの更新」とも言います。

**Pixel Diff vs Visual AI**
- **Pixel Diff**: ピクセル単位で機械的に比較（Playwright, BackstopJS など、OSSの主流）
- **Visual AI**: AIが「人間にとって意味のある差分か」を判定（Applitools などの有料ツール）

## VRTの基本手順

```
[1] ベースライン作成
    ↓
[2] コード変更
    ↓
[3] テストスナップショット取得
    ↓
[4] ベースラインと比較 → 差分画像生成
    ↓
[5] 差分があれば失敗 → 人間がレビュー
    ↓
   ├─ 意図した変更 → ベースラインを更新（Approve）
   └─ 意図しない変更 → コード修正 → [3]に戻る
```

### 詳細ステップ

**1. ベースラインの作成（初回のみ）**
正しい見た目の状態でテストを実行し、スクリーンショットを撮影。これをベースラインとしてリポジトリやクラウドストレージに保存します。

**2. テストの実行**
コード変更後、同じ条件（同じURL、同じビューポート、同じブラウザ）で再度スクリーンショットを撮影。

**3. 比較**
ベースライン画像と新しいスクリーンショットをピクセル単位で比較。しきい値を超える差分があれば「失敗」と判定。

**4. レビュー**
失敗した場合、人間が3枚の画像を見て判断します。
- **Expected**（期待）: ベースライン
- **Actual**（実際）: 今回のスクリーンショット
- **Diff**（差分）: 違いをハイライトした画像

**5. アクション**
- 差分が**バグ**なら → コードを修正してテストを再実行
- 差分が**意図した変更**なら → ベースラインを新しい画像で更新

## ベースライン運用の重要ポイント

ベースラインの扱いはVRT運用の肝になるので少し補足します。

**どこに保存するか**
- **Git で管理**: PNGファイルをリポジトリにコミット（Playwright のデフォルト）。小〜中規模ならこれで十分。
- **クラウドストレージ**: S3 や GCS に保存（reg-suit などはこのパターン）。リポジトリが肥大化しない。
- **SaaS のクラウド**: Chromatic などが内部で管理。

**いつ・どこで生成するか**
ここが一番揉めます。ローカルマシン（macOS）と CI（Linux）ではフォントレンダリングが違うため、ローカルで生成したベースラインは CI で必ず失敗します。

実務的な解は **「ベースラインは CI 上でのみ生成する」** で、開発者は `--update-snapshots` をローカルで実行せず、PR にラベルを付けたら GitHub Actions がベースラインを再生成してコミットを返してくる、といったフローを組みます。

**誰が承認するか**
ベースライン更新は「見た目の仕様変更を受け入れる」という意思決定なので、PR レビューと同じくチームで承認する運用が安全です。Chromatic などの SaaS は専用のレビューUIを提供しますが、OSSベースなら PR の画像差分をそのままGitHub上で確認します（GitHub は画像のswipe diffに対応しています）。

## ざっくり対比表

| 用語 | 意味 | 別名 |
|---|---|---|
| Baseline | 正解として保存された基準画像 | Golden image, Reference image |
| Snapshot | 撮影された画像（文脈で意味が変わる） | Screenshot |
| Diff | ベースラインとの差分を可視化した画像 | Difference image |
| Threshold | 許容する差分の上限 | Tolerance |
| Flaky | 結果が不安定なテスト | - |
| False Positive | 偽の検知（バグでないのに失敗） | 偽陽性 |
| Approve | 差分を意図した変更として受け入れる | Accept, Update baseline |
| Masking | 動的領域を比較対象外にする | Ignore region |

---

これで概念は一通り押さえられたと思います。実装の細かい部分（例えば「play関数完了の待ち方」「Docker でのフォント統一」など）で深掘りしたいところがあれば続けてください。
